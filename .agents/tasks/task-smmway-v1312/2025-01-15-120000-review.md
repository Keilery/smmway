# SMMWay v1.3.12 ‚Äî Chat commands, order safety, and lot matching fix

This release adds buyer-facing chat commands (`!—Å—Ç–∞—Ç—É—Å`, `!—Ä–µ—Ñ–∏–ª–ª`, `!–æ—Ç–º–µ–Ω–∞`), hardens order processing against missing attributes on FPC order objects, fixes bidirectional lot-title matching that previously missed partial overlaps, raises the minimum auto-price floor from 0 to 1.0, removes `#smmway` hashtags from lot descriptions, and documents the new commands in lot templates.

**Watch for:** The `!–æ—Ç–º–µ–Ω–∞` command triggers a FunPay refund to the buyer based on heuristic parsing of the SMM API cancel response (confirmed) ‚Äî a false positive on the word "cancel" in the status field could cause unintended refunds. The bidirectional title match (`norm in t_norm`) can false-match short lot descriptions against unrelated orders (likely). The ETA calculation in `!—Å—Ç–∞—Ç—É—Å` uses `start_count + quantity - remains` which produces wrong results when the SMM API's `start_count` already accounts for progress (possible).

## High-level view

Order processing now wraps every attribute access on the FPC order object in `getattr` with fallbacks. If no usable `chat_id` can be resolved, the order is skipped with a Telegram notification ‚Äî fail-closed. However, the `chat_id` fallback chain (`buyer_id` ‚Üí `buyer_node_id`) may produce a valid integer that isn't actually a usable FunPay chat, leaving the order stuck in `awaiting_link`.

Three new buyer chat commands give end-users self-service capabilities: status polling with ETA estimation, refill requests (gated by the service's `refill` flag), and order cancellation with automatic FunPay refund when the SMM platform confirms. The cancel path is the most consequential because it moves money ‚Äî refund logic hinges on substring matching of the API response which could misfire.

Lot matching gained a `#lot:123` marker lookup and bidirectional substring matching (lot title in description OR description in lot title). The bidirectional approach fixes cases where FunPay truncates or extends the description, but it widens the match surface and could produce false positives on very short titles.

The minimum lot price floor moved from 0.0 to 1.0, but the config comment still says "–ø–æ–ª –ø–æ-—É–º–æ–ª—á–∞–Ω–∏—é —É–±—Ä–∞–Ω" ‚Äî contradictory documentation.

<summary>Issues (5)</summary>

1. **Cancel refund heuristic** ‚Äî `_handle_cancel_command` triggers FunPay refund if the string "cancel" appears anywhere in the SMM response status field. Use an explicit success indicator (e.g., check for an exact status value or a dedicated boolean field) instead of substring matching.
2. **Bidirectional match false positives** ‚Äî `find_lot_by_title` matches if the full description is a substring of the lot title (`norm in t_norm`). A short description like "VK" would match any lot with "VK" in its title. Add a minimum-length guard or prefer the `#lot:` / `[#NNN]` markers.
3. **ETA calculation assumption** ‚Äî `_handle_status_command` computes `done = start_count + quantity - remains`, assuming `start_count` is a pre-order baseline. If the SMM API returns `start_count` as the current counter, the ETA will be wildly off. Validate or document the expected semantics.
4. **No authorization on chat commands** ‚Äî Any user in a chat can issue `!—Ä–µ—Ñ–∏–ª–ª` or `!–æ—Ç–º–µ–Ω–∞` and it operates on the latest order for `msg.author_id`. If FPC surfaces messages from non-buyers in the same chat thread, commands could target the wrong order.
5. **Stale order state in cancel** ‚Äî `_handle_cancel_command` finds the order in local storage without refreshing its SMM status first. If the order completed between the last status poll and the cancel request, the local `status != "completed"` check passes but the cancel API call fails or cancels a completed order.

## Safe attribute access in `on_new_order`

The `chat_id` resolution chain falls back through `getattr(order, "chat_id", None) or getattr(order, "buyer_id", None) or getattr(order, "buyer_node_id", None)`. If `buyer_id` is a valid integer but not actually usable as a FunPay chat identifier, the plugin will attempt to send messages to an ID that isn't a real chat ‚Äî the message silently fails via the exception handler in `send_buyer_message`, but the order sits in `awaiting_link` forever with no way to reach the buyer and no Telegram alert (the `if not chat_id` guard only fires on zero/None).

## `!–æ—Ç–º–µ–Ω–∞` and the refund decision

```python
refunded = False
if isinstance(cancel_status, str) and "cancel" in cancel_status.lower():
    refunded = True
elif result.get("refund"):
    refunded = True
```

The refund flag is derived from two heuristics: (1) the word "cancel" appears somewhere in the status string returned by the SMM API, or (2) the response has a truthy `refund` key. Heuristic (1) is fragile ‚Äî an API response like `{"status": "Partial. Cannot cancel."}` would set `refunded = True` and trigger a FunPay refund to the buyer even though the cancellation failed. This is a money-losing failure mode.

There's no verification that the refund amount matches what was charged ‚Äî a partial cancel on the SMM side could lead to a full refund on FunPay. Consider checking for an exact status value (e.g., `status == "Cancelled"`) rather than a substring match, and comparing the `charge` field against the FunPay price before initiating refund.

## Bidirectional title matching in `find_lot_by_title`

The new bidirectional check adds `norm in t_norm` (description is a substring of the lot title). If the incoming `description` is something short ‚Äî a service name fragment like "–ª–∞–π–∫–∏" ‚Äî it will match the first lot whose title contains that word. The `#lot:123` and `[#NNN]` markers are checked first, so this only fires when markers are absent. In production with dozens of lots sharing similar keywords, the first iteration-order match wins, which may not be the correct lot.

A minimum-length threshold on `norm` before allowing the `norm in t_norm` branch (e.g., require at least 10 characters) would reduce false positives without breaking the truncation-recovery use case.

## `!—Å—Ç–∞—Ç—É—Å` ETA estimation

```python
done = int(start_count) + found.quantity - int(remains)
if done > 0 and int(remains) > 0:
    rate_per_sec = done / max(elapsed_sec, 1)
    eta_sec = int(int(remains) / rate_per_sec)
```

The formula assumes `start_count` is a snapshot of the counter *before* the order began. This matches common SMM panel API semantics, but not all providers use this convention. The entire block is wrapped in `except Exception: pass`, so a wrong calculation suppresses the ETA silently ‚Äî no user-visible breakage, but a buyer could also receive a wildly inaccurate ETA if the types parse correctly but the semantics are wrong.
